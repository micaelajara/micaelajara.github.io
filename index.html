01

El testing sin contexto es disparar al aire. Antes de armar un caso de prueba, entiendo qué flujo protege y qué impacto tiene si falla.

03

Me muevo entre lo técnico y lo humano

QA
Prelaunch QA & regresión QA▼
En Penske Truck Rental era responsable del QA antes y después de cada deploy. Hacía regresión en las funcionalidades críticas del sitio, especialmente en las áreas que tocaba cada cambio, y comparaba contra producción para asegurarme de que nada se hubiera roto. Identifiqué varios bugs severos antes de que llegaran a los usuarios. At Penske Truck Rental I was responsible for QA before and after every deploy. I ran regression on critical site features, especially areas touched by each change, and compared against production to make sure nothing broke. I caught several severe bugs before they reached users.
Testing de features nuevas y redesigns QA▼
Cubrí el testing end-to-end de features nuevas en Penske, desde revisar los criterios de aceptación en los tickets hasta validar la implementación final. En los redesigns comparaba la implementación contra el Figma: colores, tipografía, espaciado, comportamiento responsive. Una cosa es que funcione, otra es que se vea como se especificó. I covered end-to-end testing of new features at Penske, from reviewing acceptance criteria in tickets to validating the final implementation. For redesigns, I compared the build against Figma: colors, typography, spacing, responsive behavior. Working is not the same as looking right.
Reporte de bugs QA▼
Mis reportes siempre incluyen título claro, ambiente, pasos para reproducir, resultado actual vs esperado, severidad y evidencia visual. En Penske esto era crítico porque el código personalizado del cliente hacía que los bugs fueran difíciles de reproducir para el equipo de desarrollo. Con un buen reporte, el tiempo de resolución se reducía significativamente. My bug reports always include a clear title, environment, steps to reproduce, actual vs expected result, severity, and visual evidence. At Penske this was critical because the client's custom code made bugs hard to reproduce for the dev team. A good report meant faster resolution.
Testing exploratorio QA▼
Gran parte de mi trabajo en RebelMouse era exploratorio aunque no lo llamara así: navegaba el sitio como un usuario real, siguiendo flujos que nadie había documentado como caso de prueba. Encontré bugs que los test cases formales no habrían cubierto, sobre todo en las intersecciones entre el código del cliente y las features nuevas de la plataforma. Much of my work at RebelMouse was exploratory even before I had a name for it: I navigated the site as a real user, following flows no one had documented as test cases. I found bugs that formal test cases would have missed, especially at the intersection of client code and new platform features.
Priorización bajo presión QA▼
Con tiempos ajustados antes de cada deploy, siempre empezaba por lo que más impacto tenía en el negocio: funcionalidades core del sitio, flujos de usuario críticos, y las áreas que tocaba el cambio. En un cliente enterprise como Penske, una hora con algo roto era una hora con impacto real en su operación. With tight timelines before each deploy, I always started with what mattered most to the business: core site features, critical user flows, and areas touched by the change. For an enterprise client like Penske, one hour with something broken was one hour of real operational impact.
Herramientas
Postman & testing de APIs Tools▼
Uso Postman para testear APIs REST, verifico status codes, el body de la respuesta y que la información del JSON sea correcta. También valido autenticación: si una acción devuelve 401 o 403, el problema es distinto y el fix es diferente. Uso variables de entorno para testear contra distintos ambientes sin modificar cada request a mano. I use Postman to test REST APIs, verifying status codes, response body, and JSON data accuracy. I also validate authentication: a 401 and a 403 mean different things and require different fixes. I use environment variables to test across environments without modifying each request manually.

DevTools era mi herramienta diaria en Penske para inspeccionar requests, analizar respuestas de API, detectar errores de consola y revisar cómo se cargaban los recursos del sitio. Cuando el sitio tenía problemas de rendimiento, la pestaña de red era el primer lugar donde miraba. DevTools was my daily tool at Penske for inspecting requests, analyzing API responses, catching console errors, and reviewing how site resources loaded. When the site had performance issues, the network tab was the first place I looked.
Jira & gestión de bugs Tools▼

También seguía el lifecycle de cada bug: si volvía como Fixed, lo verificaba yo antes de cerrarlo. Si no estaba resuelto, lo reabrí con evidencia. I tracked all bugs in Jira, writing tickets with enough context for devs to reproduce them without back-and-forth. I also followed each bug's lifecycle: if it came back as Fixed, I verified it before closing. If it wasn't resolved, I reopened it with evidence.

I'm adding SQL to my testing toolkit to verify data directly in the database after an action in the app. I work with SELECT, WHERE, and JOIN, enough to validate data without depending on someone else to show me.

Proceso
Mejora de procesos con el cliente Proceso▼
En Penske mejoré la forma en que le mostrábamos los test cases al cliente. Antes el proceso era poco claro, el cliente no sabía qué se había testeado y qué no. Implementé una forma de presentar la cobertura de testing que daba visibilidad real: qué features se cubrieron, qué quedó fuera y por qué. Eso generó más confianza y menos idas y vueltas. At Penske I improved how we presented test cases to the client. The previous process was unclear, the client didn't know what had been tested and what hadn't. I implemented a coverage presentation that gave real visibility: what features were covered, what was left out, and why. That built trust and reduced back-and-forth.
Test design aplicado Proceso▼
Para decidir qué casos testear uso equivalence partitioning, agrupar inputs que se comportan igual y testear uno por clase, en lugar de repetir casos redundantes, y boundary value analysis para los límites, donde suelen aparecer los bugs. En formularios y validaciones de Penske esto me permitía cubrir más con menos casos. To decide what to test I use equivalence partitioning, grouping inputs that behave the same and testing one per class instead of repeating redundant cases, and boundary value analysis for the edges, where bugs tend to show up. In Penske's forms and validations, this let me cover more with fewer test cases.
Trabajo con equipos de desarrollo Proceso▼
En RebelMouse trabajaba directamente con el equipo de desarrollo, reportaba bugs, los discutía y los escalaba cuando era necesario. Cuando un dev decía que algo no era un bug, pedía que me mostraran cómo lo reproducían. A veces era un tema de ambiente o datos de prueba. Si no, escalaba con evidencia. El objetivo siempre fue resolver el problema, no tener razón. At RebelMouse I worked directly with the dev team, reporting bugs, discussing them, and escalating when needed. When a dev said something wasn't a bug, I asked them to show me how they reproduced it. Sometimes it was an environment or test data issue. If not, I escalated with evidence. The goal was always to solve the problem, not to be right.
