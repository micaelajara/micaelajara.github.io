Bloque 1 — Fundamentos de QA
¿Cuál es la diferencia entre SDLC y STLC? B1Exp. ▼
El SDLC es el ciclo completo de desarrollo del software — desde la idea hasta el mantenimiento. El STLC es el ciclo específico de testing, que incluye planificación, diseño de casos, ejecución, reporte de defectos y cierre. El QA entra desde la fase de requisitos del SDLC, no cuando ya hay algo que probar. Eso permite detectar inconsistencias o ambigüedades en los requisitos antes de que lleguen al desarrollo, lo que reduce el costo de corrección.
Contame sobre un desafío de QA que hayas enfrentado y cómo lo resolviste. B1Exp. ▼
En RebelMouse trabajaba con clientes enterprise que tenían código propio integrado a la plataforma — Penske Truck Rental era el más grande. Cuando salían features nuevas, ese código personalizado a veces generaba bugs en producción que el equipo de desarrollo no detectaba porque no conocían el contexto de cada cliente. Mi rol era hacer regresión rápida en los clientes más críticos antes y después de cada deploy. Varias veces identifiqué bugs severos en producción y los reporté con suficiente detalle como para que desarrollo pudiera reproducirlos y resolverlos rápido. Eso reducía el tiempo que un cliente enterprise estaba con algo roto — cualquier hora de downtime tenía impacto directo en su negocio.
¿Cómo priorizás qué testear cuando el tiempo es limitado? B1Exp. ▼
Priorizo por impacto en el negocio. Primero las funcionalidades críticas — las que si fallan afectan directamente al usuario o al cliente. Después regresión en las áreas que tocó el cambio, y si queda tiempo, el resto. En Penske lo hacía así constantemente: era un cliente enterprise y no había margen para que algo core estuviera roto en producción. También mejoré el proceso de mostrar los test cases al cliente para que tuvieran visibilidad de qué se había cubierto y qué no.
¿Cuál es la diferencia entre testing de regresión, smoke y sanity? B1Exp. ▼
¿Cuándo?¿Qué cubre?Velocidad
SmokeAntes de testear en profundidadLo más básico del sistemaMuy rápido
SanityDespués de un fix puntualSolo esa áreaRápido
RegresiónDespués de un deployTodo lo que ya funcionabaMás completo
Regresión es para verificar que un deploy no rompió funcionalidad existente. Smoke es una prueba rápida para confirmar que el sistema está estable antes de testear en profundidad. Sanity es más acotada — la uso cuando hay un fix específico para verificar que ese fix funciona y no afectó lo que está alrededor. En RebelMouse después de cada fix hacía sanity para validar el cambio puntual, y después un smoke rápido comparando contra producción.
¿Qué información no puede faltar en un buen reporte de bug? B1Exp. ▼
Un buen reporte de bug tiene que tener título claro, ambiente, dispositivo y OS, pasos para reproducir, resultado actual vs resultado esperado, severidad y evidencia — screenshots o video. Sin los pasos para reproducir, desarrollo pierde tiempo tratando de encontrar el bug. Sin la severidad, no se puede priorizar.
¿Cómo manejás la relación con desarrollo cuando un dev dice que el bug no es un bug? B1Exp. ▼
Primero pido que me muestren cómo lo reproducen ellos, porque a veces es un tema de ambiente o datos de prueba distintos. Si seguimos en desacuerdo, escalo con evidencia — los pasos exactos, el ambiente, screenshots — y si es necesario lo llevamos a producto o al negocio para decidir si el comportamiento es el esperado. El objetivo no es tener razón sino que el usuario tenga una buena experiencia.
¿Cuál es el rol del QA en un equipo ágil? ¿Participás en las ceremonias? B1Exp. ▼
En RebelMouse participaba principalmente de la daily por cómo estaba organizado el equipo. Pero entiendo el rol del QA en cada ceremonia — en el planning para revisar criterios de aceptación, en la review para validar lo entregado, y en la retro para aportar desde la perspectiva de calidad. Es algo que quiero desarrollar más en mi próximo rol.

¿Qué es un test plan y qué contiene? B1Exp. ▼
Un test plan es el documento que define cómo se va a enfocar el testing de un producto o feature. Incluye el alcance — qué se testea y qué queda fuera —, los objetivos, los tipos de testing a ejecutar, los criterios de salida y los riesgos identificados. En RebelMouse el proceso era ágil e informal, pero siempre definía el alcance antes de testear — qué features cubrir, qué quedaba fuera por tiempo, en qué ambiente y qué criterio usaba para considerar algo listo.
¿Cómo explicarías la diferencia entre testing funcional y testing exploratorio? B1Exp. ▼
El testing funcional valida que una feature cumple con los requisitos — tenés casos definidos y los ejecutás. El exploratorio es sin guión: explorás el producto como un usuario, confiando en tu criterio para encontrar bugs que los casos formales no cubren. En la práctica uso los dos — funcional para tener cobertura sistemática, exploratorio para encontrar lo que nadie pensó en documentar.
¿Qué es el bug lifecycle? Describí los estados por los que pasa un bug. B1Exp. ▼
Un bug pasa por varios estados: se crea como New, se asigna a un dev, pasa a In Progress mientras lo trabajan, luego a Fixed o Ready for Review. El QA lo verifica — si está resuelto se cierra, si no se reabre. También puede ser rechazado si desarrollo determina que es comportamiento esperado.

New → Assigned → In Progress → Fixed / Ready for Review → Verified → Closed / Reopened / Rejected

¿Cómo diferenciás la severidad de la prioridad en un bug? B1Exp. ▼
Son conceptos distintos. La severidad es el impacto técnico del bug — qué tan roto está algo. La prioridad es qué tan urgente es resolverlo según el negocio. Un bug puede ser severo pero de baja prioridad si afecta una funcionalidad poco crítica, o al revés — algo menor visualmente pero de alta prioridad si está en un lugar muy visible para el usuario.
¿Qué harías si encontrás un bug crítico el día antes de un release? B1Exp. ▼
Lo reporto de inmediato con toda la evidencia — severidad, impacto, pasos para reproducir — y lo escalo al equipo y a producto. No es una decisión que tomo sola: presento el riesgo claramente y dejo que el negocio decida si posponer el release, sacar la feature con un flag o asumir el riesgo. Mi rol es asegurarme de que tomen la decisión con información completa.
Bloque 2 — Testing técnico
¿Qué es una API REST y cómo la testearías? B2Exp. ▼
Una API REST es una interfaz que permite la comunicación entre sistemas usando HTTP. Para testearla uso Postman o DevTools — verifico el status code, el body de la respuesta y que la información del JSON sea correcta. También valido autenticación: un 401 significa que el usuario no está autenticado, un 403 que está autenticado pero no tiene permisos para esa acción.
¿Qué son las variables de entorno en Postman y para qué las usarías? B2Exp. ▼
Las variables de entorno en Postman permiten guardar valores que cambian según el ambiente — como la URL base o un token de autenticación — y reutilizarlos en todos los requests usando {{nombre_variable}}. Esto evita modificar cada request manualmente cuando cambiás de staging a producción. Las usé para testear APIs en distintos ambientes.

Ejemplo: en lugar de escribir la URL completa en cada request, guardás {{base_url}} y cambiás solo el environment desde el dropdown.

¿Qué son los scripts de test en Postman y para qué sirven? B2Aprendiendo ▼
En Postman se pueden escribir scripts de test en la pestaña Tests para automatizar la verificación de respuestas — status code, que existan ciertos campos en el body, valores específicos en el JSON. Es la diferencia entre testing manual de APIs y testing automatizado dentro de Postman: corrés toda una colección de requests y ves de un vistazo cuáles pasaron y cuáles fallaron.
pm.test("Status code es 200", function () { pm.response.to.have.status(200); }); pm.test("La respuesta tiene nombre", function () { var body = pm.response.json(); pm.expect(body.name).to.exist; });
¿Qué es el performance testing? ¿Cuál es la diferencia entre load, stress y spike? B2Aprendiendo ▼
El performance testing evalúa cómo se comporta un sistema bajo carga. Hay tres tipos principales: load testing simula la carga normal esperada; stress testing lleva el sistema más allá de su límite para ver en qué punto falla; y spike testing simula un pico repentino de usuarios.

¿Para qué usás SQL en testing? ¿Qué comandos conocés? B2Aprendiendo ▼
SQL es el lenguaje para consultar bases de datos relacionales. Como QA lo uso para verificar datos — después de ejecutar una acción en la app, consulto la base de datos para confirmar que los datos se guardaron correctamente. Los comandos principales son SELECT para traer datos, WHERE para filtrar por condición, y JOIN para combinar datos de dos tablas.
-- Traer todos los usuarios SELECT * FROM users; -- Filtrar por condición SELECT * FROM users WHERE email = 'test@test.com'; -- Combinar dos tablas SELECT users.name, orders.product FROM users JOIN orders ON users.id = orders.user_id;
¿Qué es el visual testing? B2Exp. ▼
El visual testing verifica que la implementación visual coincida con el diseño — mockup, Figma o guía de estilos. Se revisan colores, tipografía, espaciado, alineación y comportamiento responsive. Una feature puede funcionar correctamente pero verse diferente a lo que diseño especificó. En Penske lo apliqué especialmente en los redesigns — comparar la implementación contra el Figma era parte del proceso de prelaunch QA.
¿Qué es el test design? ¿Qué técnicas conocés? B2Exp. ▼
El test design son las técnicas para decidir qué casos de prueba crear de manera sistemática. Las dos principales son equivalence partitioning — agrupar inputs en clases que se comportan igual y testear uno por clase — y boundary value analysis, que testea los valores exactos en los límites del rango, donde suelen aparecer los bugs. En la práctica se usan juntas.
